- 盘点范围：主 Flask 应用、已注册蓝图、部门权限配置、用户名白名单、数据范围限制、本地开发账号、飞书授权及独立子服务

## 一、权限体系总览

项目当前不是单一权限系统，而是以下几类规则共同决定最终访问结果：

1. **登录身份**：主要使用飞书用户 ID、用户名和部门信息；部分接口也识别飞书请求头。
2. **统一功能权限**：由 `department_permissions.py` 中的 `PERMISSION_CONFIG` 和 `@require_permission(...)` 控制。
3. **部门树继承**：`内容生产工厂`、`运营部`可向下继承到子部门；`新人组`被明确排除在部门树继承之外。
4. **用户名白名单**：部分 TK 看板、创新委员会、报告中心等功能按姓名单独放行。
5. **数据范围权限**：即使能进入页面，也可能只能看本人、本部门、指定运营部门或指定店铺的数据。
6. **模块内手工校验**：部分模块没有使用统一装饰器，而是在路由函数内部检查登录、姓名或部门。
7. **开发账号绕过**：配置开发者和本地 `dev_*` 账号存在单独的模拟权限。
8. **外部平台授权**：飞书通讯录、消息、云文档、审批等还受飞书应用 Scope 约束。

> “菜单能看到”和“后端接口能访问”不是同一件事。前端卡片通常根据统一权限显示，但后端是否真正拦截，要看对应路由是否使用权限装饰器或手工校验。

### `all` 的含义

权限配置中的 `all` 表示**所有已登录用户**，不表示匿名公网访问。

### 多部门用户

用户同时属于多个有效部门时，当前规则采用并集：任一部门满足条件即可获得对应功能权限。无法识别或未映射的部门会被忽略，系统配置了飞书 API 回退查询。

### 部门树规则

- 可继承根部门：`内容生产工厂`、`运营部`。
- 明确排除继承：`新人组`。
- 因此新人组只获得权限表中**明确写入“新人组”**的权限，不会因为组织树位置自动继承内容生产工厂或运营部的其他权限。
- 当前权限配置里还出现了未写入硬编码映射表的名称：`运营部`、`TK直播部`、`深圳团队`、`亚马逊`、`质检`、`文员`、`桌长`、`库管`。这些名称需要依赖飞书 API 回退、实际父子部门信息或其他运行时数据识别。
- `tk_project_group` 的允许部门数组中还存在一个空字符串 `""`，当前没有明确业务含义。

### AI 部权限调试

- 权限调试入口本身需要 `ai_dept`，只有真实部门包含 AI 部的用户才能启用模拟。
- 可按指定部门或指定用户的实际部门模拟首页和功能权限。
- `ai_dept`、`admin_functions` 始终按真实部门判断，不受模拟部门影响，避免调试后把调试入口或管理入口锁死。
- 调试状态保存在当前会话，只影响当前登录会话，不会修改任何人的正式权限。

## 八、模特库与排队数据权限

- 模特库页面：需要 `model_library`。
- 模特排队页面：需要 `model_queue`，该权限对全体已登录用户开放。
- `/api/models` 查询通常按“上传人 = 当前用户名”过滤，即普通用户看自己上传的模特。
- “我的排队”操作通常按“创建人 = 当前用户名”过滤。
- `/api/models_all` 返回全量模特数据。
- 模特文件、模型增删改、排队总览及部分完成/取消接口没有全部绑定 `model_library` 或 `model_queue` 装饰器。
- 部分按排队 ID 执行的全局完成/取消操作未发现创建人校验，不能只依靠页面入口权限判断安全边界。

## 九、图片工具与独立子服务权限

视觉设计板块的统一权限 `visual_design_dept` 对全体已登录用户开放。Flask 主站会生成带签名的图片工具访问参数：

- 用户参数：`auth_user`
- 签名参数：`auth_token`
- 签名盐标识：`lashforge-auth-v1`

独立图片工具当前规则：

- `周俊成`可查看所有账号的生成历史。
- 其他用户按当前账号查看自己的历史。
- 没有有效用户签名时，独立服务代码会落到“访客”身份，因此独立服务本身不是严格的“无签名即拒绝”模式。

Seedance Web 模块采用“已登录即可访问”的规则，没有单独部门权限。

## 十一、开发者与本地调试权限

### 配置开发者

当前开发者模式开启：

- 开发者用户 ID：`ou_developer_id`
- 开发者邮箱：`developer@company.com`
- `bypass_all_permissions = True`

命中配置开发者后可绕过全部权限。

### 本地 `dev_*` 账号

本地开发账号在“菜单权限计算”和“路由装饰器校验”中存在两套不完全一致的规则：

| 本地账号 | 菜单/可访问功能计算 | 路由装饰器中的放行 |
|---|---|---|
| `dev_bd` | 创新提案、内容生产工厂、脚本生成、达人管理 | 差评分析、创新提案、脚本生成、达人管理 |
| `dev_video` | 创新提案、内容生产工厂、脚本生成、达人管理 | 差评分析、创新提案、脚本生成 |
| `dev_warehouse` | 创新提案、仓储部组 | 未配置对应装饰器放行 |
| `dev_operation1/2/3/6` | 创新提案、差评分析、对应运营板块、客服 | 未配置对应装饰器放行 |
| `dev_ai` | 全部 | 差评分析、创新提案、管理员功能 |
| `dev_admin` | 全部 | 全部 |
| 其他 `dev_*` | 仅创新提案 | 装饰器默认不放行 |

因此本地调试时可能出现“菜单可见但接口 403”或“接口可进但菜单不显示”的情况。

## 十二、管理后台权限变更能力

管理后台的权限管理页面和授权/撤销接口受 `admin_functions` 控制，即总经办、AI部或开发者可进入。

但当前代码中的“给单个用户授权/撤销”没有持久化存储实现：

- 接口可能返回操作成功。
- 会清理相关权限缓存。
- 重新计算时仍主要依据部门和硬编码规则。
- 因此它目前不能作为可靠的长期个人授权机制。

## 十三、飞书应用授权 Scope

以下属于飞书平台授权，不是项目内部业务角色，但会决定相关功能能否正常调用。

### 用户登录授权请求

- `contact:user.id:readonly`
- `im:chat`
- `im:chat:read`
- `im:chat:readonly`
- `im:message:readonly`
- `im:message`
- `im:message.group_msg:get_as_user`
- `im:message.p2p_msg:get_as_user`

### 代码中涉及的应用能力

- 通讯录：用户、部门、部门负责人和组织关系读取。
- 即时消息：单聊/群聊消息读取和发送。
- 云文档/知识库：文档、Wiki 节点、Drive 文件和文档媒体读取。
- 审批：审批实例、任务和相关数据读取。
- 事件回调：配置了验证 Token 时，回调 Token 不匹配会拒绝请求。

飞书后台还需为应用配置与上述调用匹配的应用权限，例如 Wiki、文档媒体下载、Drive/Docx 读取、通讯录、消息和审批权限。

## 十四、当前权限绑定缺口与注意事项

以下仅记录现状，本次未修改：

| 级别 | 现状 |
|---|---|
| 配置未绑定 | `tk_dashboard`、`warehouse_group`、`rd_dept` 未发现当前路由直接使用对应权限键 |
| 性能监控 | `performance_monitor` 已配置为 AI部/总经办，但 `/performance` 当前只要求登录 |
| 缓存管理 | `/api/cache/stats`、`/api/cache/clear` 未发现模块权限装饰器；其中清理接口会修改全局缓存状态 |
| 差评分析 | 主页面、分析、词频受控，但历史和详情接口未统一绑定 `review_analysis` |
| 达人管理 | 主入口和主查询受控，部分标签、编辑、删除、支付和筛选接口未统一绑定 |
| 模特管理 | 页面受控，但模型文件、全量查询及多项增删改/排队接口未全部绑定 |
| 创新提案 | 统一权限主要保护入口，蓝图内部多项接口依赖模块内零散校验 |
| 培训内容 | `/training_videos/rococo` 及对应文件路由未发现显式登录检查 |
| 权限后台 | 单用户授权/撤销没有持久化，不能形成长期权限 |
| 本地开发 | 菜单权限与路由装饰器规则不一致 |
| 部门配置 | 多个允许部门未写入硬编码映射，依赖 API 回退；`tk_project_group` 中存在空字符串 |
| 姓名白名单 | 多处直接按中文姓名判断，人员改名、重名或错别字都会影响权限 |
| 前后端一致性 | 卡片隐藏不能替代后端鉴权，部分模块存在入口受控而子接口未统一受控的情况 |
